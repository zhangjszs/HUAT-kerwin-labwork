## 核心项目一：迷宫寻路系统（算法基础）

**技术栈：** C++, Qt 6.7.2, CMake

**核心实现：** 独立设计并实现了包含 A*、BFS、DFS、Dijkstra、JPS 五种寻路算法的可视化系统，支持4/8方向移动、多种启发函数切换、实时步骤演示和算法性能对比。

**个人贡献：** 独立完成系统架构设计、全部算法实现（约1,666行核心代码）、Qt GUI 交互设计。

**关键决策：**
选择 A* 作为默认算法，而非 BFS 或 Dijkstra，原因是：在等权网格地图上，A* 通过启发函数引导搜索方向，在保证最优解的前提下将搜索节点数减少约33%（实测：A* ~800节点 vs BFS ~1200节点）。JPS 进一步利用网格对称性，在开阔地图上可减少60-80%的搜索节点。

**项目地址：** `data-structures-course-design/coding/MazeProject/`

## 核心项目二：Titanic 生存预测（数据科学实践）

**技术栈：** Python, Jupyter Notebook, pandas, scikit-learn, XGBoost

**核心实现：** 完整的端到端数据科学项目，覆盖问题定义 → EDA → 特征工程 → 多模型对比 → 模型解释 → 结论反思全流程。

**个人贡献：** 独立完成全部分析，每个步骤均记录决策依据而非仅记录操作。

**关键决策：**
1. **评估指标选择：** 选用 F1-score 而非 Accuracy，因数据集类别不平衡（约62%死亡），Accuracy 会产生误导性结论。
2. **特征工程：** age 缺失用中位数而非均值填充（年龄分布右偏，中位数更稳健）；构造 family_size 特征验证"独行旅客 vs 家庭"假设。
3. **模型选型：** 逻辑回归作为可解释基线，随机森林提供特征重要性，XGBoost 追求最优性能——三者形成完整的对比框架。

**主要结论：** XGBoost 表现最优（Test AUC ~0.87），最重要特征为性别、票价、年龄，与 EDA 阶段观察一致。

**项目地址：** `ml-practice/titanic-survival-prediction/`

## 技术思考

在迷宫系统的开发过程中，我深入思考了算法选择的权衡问题。JPS 相比 A* 在开阔地图上有显著优势，但其实现复杂度更高，且在障碍物密集的迷宫中优势减弱。这让我意识到：算法选择不是"哪个更好"，而是"在什么场景下哪个更合适"——这种场景化思维是工程实践中最重要的能力之一。

在 Titanic 项目中，我刻意在每个步骤记录"为什么这样做"，而不只是"做了什么"。这种习惯来自于对科研思维的理解：好的分析需要可重现、可解释，每个决策都应有据可查。
